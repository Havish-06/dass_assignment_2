\section{Introduction}
This report outlines the evaluative integration testing and the architectural call graph designed for the StreetRace Manager command-line system. The implemented system perfectly models an underground street racing ecosystem consisting of six core modules (Registration, Crew Management, Inventory, Race Management, Results, and Mission Planning) alongside two additional architectural modules specifically designed to handle robust game states.

\subsection{Additional Modules Designed}
To strictly separate concerns and enhance realism, we designed and implemented the following two additional modules:
\begin{itemize}
    \item \textbf{Maintenance Module:} Manages vehicle degradation across races. If a car is involved in an intense race, it sustains probabilistic damage. This module governs the \texttt{repair\_car()} flow (deducting cash from the Inventory) and acts as an integration gateway for Mission Planning: if a car is damaged, any planned mission automatically demands an available Mechanic.
    \item \textbf{Reporting Module:} Acts as a read-only projection over the \texttt{SystemState}. It bridges the Crew, Inventory, and Race entities to generate consolidated high-level analytics overviews (e.g., aggregate race wins, crew standings) without risking state mutations.
\end{itemize}

\section{Integration Test Design (2.2)}
Our integration test suite (\texttt{test\_streetrace\_integration.py}) actively validates how these distinctly designed modules interact. We formalized integration hooks covering all 16 major internal workflows:

\subsection{Test Scenario 1: Registering Crew and Listing Status}
\begin{itemize}
    \item \textbf{Scenario Tested:} Registering multiple crew members with distinct roles and validating they correctly populate the overall system status report.
    \item \textbf{Modules Involved:} Registration, Crew Management, Reporting.
    \item \textbf{Expected Result:} The Reporting module securely fetches the updated roster generated concurrently by Crew Management and Registration.
    \item \textbf{Actual Result:} The output report successfully joined active crew members, their availability arrays, and roles.
    \item \textbf{Errors/Logical Issues Found:} Ensured no circular dependencies occurred when the read-only reporting module accessed newly hydrated state records.
\end{itemize}

\subsection{Test Scenario 2: Adding Cars and Listing Inventory}
\begin{itemize}
    \item \textbf{Scenario Tested:} Persisting new cars through the Inventory module and extracting them uniformly via shared state constraints.
    \item \textbf{Modules Involved:} Inventory, Reporting.
    \item \textbf{Expected Result:} Cars sequentially added accurately increment the global \texttt{cars} mapping for future cross-module iteration.
    \item \textbf{Actual Result:} Perfect synchronization maintained between inventory loading and downstream availability.
    \item \textbf{Errors/Logical Issues Found:} None. Registration logic safely incremented primary keys.
\end{itemize}

\subsection{Test Scenario 3: Accumulating Spare Parts and Tools}
\begin{itemize}
    \item \textbf{Scenario Tested:} Testing Inventory module dictionaries safely accumulating aggregate quantitative integer totals.
    \item \textbf{Modules Involved:} Inventory.
    \item \textbf{Expected Result:} Sequential loads of the exact same spare part appropriately increment the mapped integer count instead of overwriting memory.
    \item \textbf{Actual Result:} Correct accumulation detected safely handling duplicate payload definitions.
    \item \textbf{Errors/Logical Issues Found:} None. 
\end{itemize}

\subsection{Test Scenario 4: Completing Races Updating Results \& Capital}
\begin{itemize}
    \item \textbf{Scenario Tested:} Running a completed race, logging results, computing random outcome distributions, and injecting prize cash securely into the ecosystem, while concurrently tracking random probabilistic vehicle damage.
    \item \textbf{Modules Involved:} Race Management, Results, Inventory, Maintenance.
    \item \textbf{Expected Result:} A generated "win" calls the Results module, which natively calls the Inventory module to add exactly $+ \$5,000$ to the overarching cash reserve. The assigned car may randomly sustain damage mapping back to Maintenance.
    \item \textbf{Actual Result:} The deterministic \texttt{mock.patch} evaluation yielded exactly $\$5,000$ cash delta properly routing across 4 separate module boundaries.
    \item \textbf{Errors/Logical Issues Found:} Fixed edge casing ensuring \texttt{race\_loss} evaluations logged exactly $0$ cash without crashing the Result delta metrics.
\end{itemize}

\subsection{Test Scenario 5: Race Selection Fails Without Eligible Driver}
\begin{itemize}
    \item \textbf{Scenario Tested:} Attempting to enter a race requiring high-skill minimums without a registered and qualified driver.
    \item \textbf{Modules Involved:} Race Management, Crew Management.
    \item \textbf{Expected Result:} The system evaluates the required minimum driver skill, detects zero available drivers fulfilling it via the cross-module Crew boundaries, and blocks race staging.
    \item \textbf{Actual Result:} The internal verification accurately raises a \texttt{ValueError} bypassing execution.
    \item \textbf{Errors/Logical Issues Found:} Initially, bounds-checking was fragile, but the tests confirmed the fail-fast rejection prevents phantom races.
\end{itemize}

\subsection{Test Scenario 6: Race Selection Fails Without Eligible Car}
\begin{itemize}
    \item \textbf{Scenario Tested:} Automatically querying available driver/car pairs when minimum vehicle speed parameters strictly outrank all active garage speeds.
    \item \textbf{Modules Involved:} Race Management, Inventory.
    \item \textbf{Expected Result:} The Race Management hooks into the Inventory module and safely triggers a failure abort when threshold parameters cannot be met.
    \item \textbf{Actual Result:} The system rejects staging the race, guaranteeing unqualified cars cannot compete.
    \item \textbf{Errors/Logical Issues Found:} Confirmed logic flawlessly parses speed limits bridging across state parameters.
\end{itemize}

\subsection{Test Scenario 7: Race Rejection for Damaged Vehicles}
\begin{itemize}
    \item \textbf{Scenario Tested:} Attempting to securely assign a car that sustained damage in a prior race into a newly scheduled race.
    \item \textbf{Modules Involved:} Race Management, Maintenance, Inventory.
    \item \textbf{Expected Result:} Race Management validates the car against the Inventory state. Finding the \texttt{damaged} flag initialized by Maintenance, it aggressively rejects the assignment.
    \item \textbf{Actual Result:} Successfully raised a \texttt{ValueError} preventing the damaged car from competing.
    \item \textbf{Errors/Logical Issues Found:} Confirmed that car state mutation perfectly syncs between Maintenance fault logging and Race vehicle selection safely.
\end{itemize}

\subsection{Test Scenario 8: Registering a Non-Driver Role for Racing}
\begin{itemize}
    \item \textbf{Scenario Tested:} Registering a driver and attempting to enter them into a competitive race. Specifically, we test registering a member structurally labeled as a \texttt{MECHANIC} and aggressively blocking them from driving.
    \item \textbf{Modules Involved:} Registration, Crew Management, Race Management.
    \item \textbf{Expected Result:} Crew registration yields success, but the Race execution module firmly blocks assigning non-driver entities to a race roster.
    \item \textbf{Actual Result:} Accurate behavior detected. \texttt{assign\_driver\_and\_car} triggers strict \texttt{Role.DRIVER} structural validation and halts execution.
    \item \textbf{Errors/Logical Issues Found:} Ensured cross-component role masking correctly flags non-drivers; no subsequent errors registered.
\end{itemize}

\subsection{Test Scenario 9: Race Loss Prevents Cash Increment}
\begin{itemize}
    \item \textbf{Scenario Tested:} Simulating a definitive street race loss and asserting that cash buffers remain entirely unmutated.
    \item \textbf{Modules Involved:} Race Management, Results, Inventory.
    \item \textbf{Expected Result:} Results logs the outcome natively, safely avoiding the downstream trigger to call Inventory cash adjustments.
    \item \textbf{Actual Result:} The cash reserve remained accurately untouched at exactly $\$0$ while tracking the final loss record reliably.
    \item \textbf{Errors/Logical Issues Found:} Validated correct boolean masking preventing phantom prize money overrides natively.
\end{itemize}

\subsection{Test Scenario 10: Damaged Cars Mandating Mechanic Availability}
\begin{itemize}
    \item \textbf{Scenario Tested:} Attempting to start a mission that explicitly requires a \texttt{MECHANIC} while a damaged car exists in the inventory, but the assigned mechanic is either missing or flagged as unavailable.
    \item \textbf{Modules Involved:} Mission Planning, Maintenance, Crew Management.
    \item \textbf{Expected Result:} The Mission module interacts with Maintenance to check for damaged cars. Since one exists, it strictly mandates that an available mechanic must exist in the crew. If none is available, the mission start is aborted.
    \item \textbf{Actual Result:} Evaluated across three vectors (no mechanic, unavailable mechanic, available mechanic). The mission accurately blocked starting until the mechanic was flagged \texttt{available=True}.
    \item \textbf{Errors/Logical Issues Found:} Ensured that cross-dependencies correctly lazy-loaded Crew states to prevent circular import failures while perfectly enforcing the business rule specified in the assignment.
\end{itemize}

\subsection{Test Scenario 11: Missions Starting Without Assigned Crew}
\begin{itemize}
    \item \textbf{Scenario Tested:} Generating a mission assignment and instantly evaluating \texttt{start\_mission} functionality without mapping minimum personnel constraints.
    \item \textbf{Modules Involved:} Mission Planning, Crew Management.
    \item \textbf{Expected Result:} The Mission blocks any state transition from \texttt{PLANNED} to \texttt{IN\_PROGRESS}.
    \item \textbf{Actual Result:} Rejection correctly observed preventing phantom data overrides over empty datasets.
    \item \textbf{Errors/Logical Issues Found:} None. 
\end{itemize}

\subsection{Test Scenario 12: Repairs Restoring Availability \& Deducting Cash}
\begin{itemize}
    \item \textbf{Scenario Tested:} Formally executing a vehicle repair via the Maintenance module natively decrementing cash inside the ecosystem.
    \item \textbf{Modules Involved:} Maintenance, Inventory.
    \item \textbf{Expected Result:} The Maintenance module flips the car state to completely operational while securely pushing a numeric deduction down into the global Inventory.
    \item \textbf{Actual Result:} Detected concurrent success: vehicle restored, integer value deducted securely matching expected targets.
    \item \textbf{Errors/Logical Issues Found:} Safely confirmed asynchronous variables could lock appropriately during state updates.
\end{itemize}

\subsection{Test Scenario 13: Repairing Vehicles Under Affordability Bounds}
\begin{itemize}
    \item \textbf{Scenario Tested:} Repairing a damaged car when the crew has an insufficient cash capital balance in the inventory.
    \item \textbf{Modules Involved:} Maintenance, Inventory.
    \item \textbf{Expected Result:} The Maintenance module attempts to bill the Inventory. Due to insufficient funds, it throws an error and crucially \textit{does not} mutate the car's damaged status.
    \item \textbf{Actual Result:} Inventory rejected the deduction, and the subsequent test assertions verified the car safely remained \texttt{damaged=True} and \texttt{available=False}.
    \item \textbf{Errors/Logical Issues Found:} Guarded against state-mutation race conditions where a car could be prematurely fixed before the cash was successfully deducted.
\end{itemize}

\subsection{Test Scenario 14: Mission Assignments \& Strict Role Verification}
\begin{itemize}
    \item \textbf{Scenario Tested:} Assigning a complex mission (e.g., heist) requiring multiple operational roles (\textbf{DRIVER}, \textbf{LEADER}) but feeding it insufficient matching crew.
    \item \textbf{Modules Involved:} Mission Planning, Crew Management.
    \item \textbf{Expected Result:} The Mission Planning module fetches current Crew statuses; detecting a missing required subset, it restricts state switching to \texttt{IN\_PROGRESS}.
    \item \textbf{Actual Result:} Assertions effectively held \texttt{MissionStatus.PLANNED}, confirming a rigid interaction hook blocking unsafe starts.
    \item \textbf{Errors/Logical Issues Found:} Validated strict subset operator matching logic across Crew data structs seamlessly. No subsequent bugs traced.
\end{itemize}

\subsection{Test Scenario 15: Maintenance Overdraws on Undamaged Hardware}
\begin{itemize}
    \item \textbf{Scenario Tested:} Directly executing \texttt{repair\_car()} on a brand new, completely undamaged car.
    \item \textbf{Modules Involved:} Maintenance, Inventory.
    \item \textbf{Expected Result:} Evaluating car integrity safely before contacting Inventory to silently drain cash capital.
    \item \textbf{Actual Result:} It safely raised an explicit \texttt{ValueError} and aborted before capital loss.
    \item \textbf{Errors/Logical Issues Found:} \textbf{Bug Encountered!} Testing confirmed the integration silently accepted the command and pointlessly deducted cash. The source logic in \texttt{maintenance.py} was refactored with an explicit `is not damaged` bailout logic clause resolving the financial drain.
\end{itemize}
